# Parallel Downloading Technique

The system follows a client-driven architecture, where the client is responsible for all scheduling and coordination, while servers remain simple and stateless.

## Client Responsibilities (Intelligent Layer)

### Metadata Retrieval
The client first fetches metadata about the file, including:
- File size
- List of available servers
- Chunk distribution across servers
- Chunk IDs and indices

### Chunk Management
The file is divided into chunks, each identified by a unique ID and index. The client keeps track of:
- Downloaded chunks
- Pending chunks
- Order of chunks for reconstruction

### Scheduling Algorithms
When multiple servers are available, the client dynamically decides which server should serve which chunk using scheduling strategies such as:
- Round Robin
- Weighted Round Robin
- Least Connections
- Least Response Time
- Resource-Based Scheduling
- Hash-Based Selection
- Random Selection
- Least Load
- Priority Scheduling
- First Come First Serve (FCFS)
- Shortest Job First (SJF)
- Multilevel Queue Scheduling
- Work Stealing
- Queue-Based Processing

### Queue Management
The client maintains multiple queues for efficient handling:

#### Main Queue
Contains all chunks that need to be downloaded and acts as the primary source of work.

#### Retry Queue
Stores chunks that failed to download, tracks which servers have already been attempted, ensures retries using different servers, and can be processed in parallel.

#### Failed Queue (Dead Letter Queue)
Contains chunks that failed after multiple retry attempts and is used for logging, alerting, or deferred retry strategies.

### Parallel Downloads
The client downloads chunks concurrently from multiple servers. Each worker independently pulls tasks from the queues, and failed downloads are automatically redirected to the retry queue.

## Server Responsibilities (Optimization Layer)
Servers are responsible for efficient chunk generation and delivery, based on their internal resources such as CPU, memory, and network bandwidth. Each server can:
- Decide optimal chunk sizes
- Optimize file access and read patterns
- Tune performance based on its capabilities

Servers do not coordinate with each other and remain stateless from a global perspective.

## Summary
The system is divided into two layers:

Client: Intelligent orchestrator responsible for scheduling, tracking, retries, and load balancing.

Server: Optimized chunk provider that is resource-aware and ensures efficient delivery.

This design ensures high performance through parallelism, scalability with multiple servers, and fault tolerance via retry and failure handling mechanisms.
